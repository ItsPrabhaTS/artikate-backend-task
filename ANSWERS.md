## Section 1 â€” Diagnose a Broken System

### Investigation log

**1. Scope the blast radius before touching code.** From APM/access logs:
which endpoints degraded, for which users, starting exactly when? Finding:
only `/api/orders/summary/`, only users with large order counts (the
`>200 orders` detail in the report is the tell), and the first slow response
lines up with the deployment timestamp. Small accounts still respond fast.
That correlation matters: latency scaling with *per-user row count* points
at per-row work, not at an overloaded box.

**2. Interrogate the deploy, not the view.** "No code change was made to
that view" only means the view *function* didn't change â€” the deploy shipped
something, and a Django view's behaviour is the product of its serializers,
model managers, middleware and dependencies. `git diff` of the release,
filtered to ORM-touching code.

**3. Distinguish the two big DB failure shapes.** Check the database slow
query log. A missing index shows up as a *few slow* queries; what I find
instead is *thousands of fast, near-identical* queries per request
(`SELECT ... FROM order_item WHERE order_id = ?` with only the parameter
changing). That shape is diagnostic: N+1, not indexing. It also explains
step 1 â€” 80ms at 20 orders, 30s at 200+ orders, because the query count is a
linear function of the user's orders.

**4. Reproduce locally and measure.** Seeded a 250-order customer
(`manage.py seed_orders`), put django-silk on the endpoint: **751 queries**
for one request.

**5. Pin the source.** The release diff shows a shared serializer gained
three fields; the view is unchanged but inherits all three costs
(`orders/serializers.py`, `BrokenOrderSummarySerializer`):
- `customer_name = CharField(source="customer.name")` â€” walks a lazy FK per
  order,
- `items = OrderItemSerializer(many=True)` â€” evaluates `order.items.all()`
  per order,
- `get_total()` â€” calls `order.items.all()` *again*, and a related manager
  returns a **fresh queryset each call**, so the previous fetch's results
  are not reused.

1 + 3N queries for N orders. Root cause category: **N+1 via ORM
lazy-loading**, introduced by serializer reuse.

### Why the fix works at the ORM and database level

The fixed view (`orders/views.py`, `OrderSummaryView`) makes the *queryset*
satisfy every field the serializer needs:

- `select_related("customer")` â€” promotes the lazy FK into a SQL `JOIN`, so
  customer columns arrive with the orders in the same round trip. Django
  hydrates `order.customer` from the row; attribute access no longer
  triggers a query.
- `prefetch_related("items")` â€” one extra query for the whole result set
  (`WHERE order_id IN (...)`), joined to parents *in Python* and stored in
  the per-instance prefetch cache. `order.items.all()` consults that cache
  instead of the database. (This is also why the broken `get_total` was
  expensive: only `.all()` reads the cache, and only when a prefetch
  populated it.)
- `annotate(total=Coalesce(Sum(F("items__unit_price") * F("items__quantity")), 0))`
  â€” moves the arithmetic into the database as a `GROUP BY` aggregate. The
  DB engine sums numbers it already has in memory; the alternative hydrates
  NĂ—M Python model instances just to add up two columns.

Result: 2 queries regardless of order count. The latency win is mostly
round trips: at ~1ms RTT the broken endpoint pays that price 751 times.

One sharp edge found while testing: adding the aggregate made Django drop
`Meta.ordering` (the ORM ignores default ordering on `GROUP BY` queries), so
the fixed queryset restates `order_by()` explicitly â€” without it the fix
would have silently changed the response order.

### Evidence

`evidence/profiling.md`, generated by `manage.py profile_summary` with
django-silk installed: **751 queries / 939ms â†’ 2 queries / 84ms** on the
same 250-order dataset. Both requests are recorded in the silk UI (`/silk/`)
for inspection. Regression guard: `orders/tests.py` asserts the fixed
endpoint runs exactly 2 queries (`django_assert_num_queries(2)`) and returns
a payload identical to the broken version's.

## Section 2 â€” SIGKILL: what happens to in-flight tasks

(Architecture and rate limiter reasoning are in DESIGN.md.)

**Celery's default is acks_early**: the worker acknowledges the message the
moment it *picks it up*, before executing. SIGKILL a worker mid-task and the
broker has nothing left to redeliver â€” the job is simply gone. That default
exists because it guarantees at-most-once, which is the safe choice for
non-idempotent tasks; for email delivery I invert it deliberately.

This implementation (`config/settings.py`):

- `task_acks_late = True` â€” the ack happens *after* the task returns. A
  worker SIGKILL'd mid-run leaves the message unacknowledged. With the Redis
  broker, kombu parks delivered-but-unacked messages in an `unacked`
  structure; once the `visibility_timeout` (here 3600s) passes without an
  ack, the message is restored to the queue and redelivered to a live
  worker. The job reruns from the start.
- `task_reject_on_worker_lost = True` â€” covers the case where the *pool
  child* is killed while the supervising worker process survives (OOM killer
  picking off a child is the common production shape). Without this flag,
  Celery marks the task failed and does **not** requeue it even with
  acks_late; with it, the message is rejected back for redelivery.
- `worker_prefetch_multiplier = 1` â€” with acks_late, every message a worker
  prefetches but hasn't started is also locked to that dead worker until the
  visibility timeout expires. Prefetch of 1 bounds that blast radius to the
  task actually executing.

**The honest consequence: at-least-once delivery.** If the worker dies
*after* the provider accepted the email but *before* the ack, redelivery
sends the email twice. Exactly-once does not exist across process
boundaries; the mitigation is an idempotency key per `message_id` honoured
by the provider. Note the interaction with the rate limiter: the first
(killed) attempt legitimately consumed a permit â€” the provider was really
called â€” and the redelivered run takes a fresh permit, which is the correct
accounting.

Tunable trade-off: `visibility_timeout` must exceed the longest legitimate
task run (or throttle wait), otherwise Redis redelivers tasks that are
still running â€” instant duplicates. 3600s is a deliberate wide margin.

## Section 3 â€” Thread-locals, async Django, and why contextvars

**Failure modes of `threading.local` tenant scoping under async Django:**

1. **One thread, many requests.** ASGI runs every request's coroutines on a
   single event-loop thread. A `threading.local` keyed by thread is
   therefore *shared by all concurrent requests*. Request A binds tenant A,
   awaits I/O; request B binds tenant B on the same thread; A resumes and
   now reads **tenant B's binding**. That is a silent cross-tenant data
   leak that only appears under concurrency â€” the worst possible failure
   mode for the "must never expose another tenant's data" requirement.
2. **Thread hops lose the binding.** Django runs sync code from async views
   via `sync_to_async` in a thread-pool executor thread. A thread-local set
   on the request thread is invisible there: the manager sees no tenant and
   (in this design) raises â€” fail-closed but broken; in a
   return-everything design it would fail *open*.

**What I would change â€” and did:** this implementation never uses
thread-locals. `tenants/context.py` uses **`contextvars.ContextVar`**.
asyncio gives each Task a copy of the current `Context`, so concurrent
requests on one thread each see their own binding, and asgiref explicitly
propagates the context across its `sync_to_async` / `async_to_sync`
boundaries, so the binding survives executor hops. The middleware is
`sync_capable`/`async_capable`, binds via `ContextVar.set()`, and resets in
a `finally` with the returned `Token` â€” cleanup is unconditional, so a
binding can never leak into the next request on a recycled worker thread.

**Honestly documented remaining bypasses** (tested or noted): `unscoped`
(the deliberate escape hatch, also `Meta.base_manager_name` â€” Django's
internals need an unfiltered manager for cascades and related descriptors),
`Model._base_manager` used directly, raw SQL / `connection.cursor()`, and
Celery tasks, which have no request and must bind explicitly with
`tenant_context(tenant)` â€” passing the tenant id in the task payload, never
implicitly.

## Section 4 â€” Written Architecture Review (Questions A and B)

### Question A â€” Django admin slow at 500k+ rows

**1. The changelist COUNT(*) â€” usually the biggest cost.** Every changelist
page runs `SELECT COUNT(*)` for the paginator, and by default a *second*
count of the unfiltered table for the "99999 total" caption
(`show_full_result_count`). On a 500k-row table (worse on Postgres, where
COUNT is a full visibility scan) that's paid on every page load. Fix: set
`show_full_result_count = False` on the `ModelAdmin`; if the paginator count
itself still hurts, override `ModelAdmin.paginator` with a class whose
`count` returns an estimate (e.g. Postgres `pg_class.reltuples`) â€” the admin
only needs it to draw page links.

**2. N+1 from `list_display` and `__str__`.** Any FK rendered in the list â€”
including one referenced inside `__str__` â€” triggers one query per row, so
100 rows Ă— 2 FKs = 200 extra queries. Fix: `list_select_related =
("customer", "product")` (JOINs them into the changelist query), or override
`ModelAdmin.get_queryset()` to add `select_related()` / `prefetch_related()`
and `.only()` to skip wide unused columns.

**3. Related widgets that enumerate the table.** A `ForeignKey` on the edit
form renders a `<select>` containing *every* related row; `list_filter` on
a related model similarly enumerates choices. With 500k rows the page
itself becomes the problem. Fix: `autocomplete_fields = ["customer"]`
(AJAX search backed by `search_fields` on the related `ModelAdmin`) or
`raw_id_fields`; for filters, prefer `RelatedOnlyFieldListFilter` or a
custom filter with a bounded choice set.

Also worth checking: `search_fields = ["name__icontains"]`-style searches
produce leading-wildcard `LIKE '%term%'`, which no B-tree index can serve â€”
on Postgres that needs a trigram GIN index â€” and `date_hierarchy` aggregates
dates across the table, so the column must be indexed. Trade-off note:
estimated counts and autocompletes sacrifice exactness and browsability for
latency â€” the right trade for an ops tool.

### Question B â€” Offset vs cursor pagination for 10,000 records

**Offset (`LimitOffsetPagination` / `PageNumberPagination` in DRF)** maps to
`LIMIT 20 OFFSET 9980`. The database cannot "skip to" an offset: it walks
and discards 9,980 rows first, so cost grows linearly with page depth, and
each page also pays the `COUNT(*)` for `count`/page numbers. The sharper
problem is **mutation during pagination**: rows are addressed by position,
so an insert at the top between page fetches shifts everything â€” a mobile
infinite scroll sees the last item of page N again at the top of page N+1
(duplicates), and deletes cause silent skips. Its strengths: random access
("jump to page 40"), a total count, trivially stateless.

**Cursor (`CursorPagination` in DRF)** addresses rows by *value*, not
position: the cursor encodes the last row's ordering key and the next page
is `WHERE (created, id) < (:last_created, :last_id) ORDER BY created DESC,
id DESC LIMIT 20` â€” a keyset seek. With a composite index on
`(created, id)` that's an index descent plus 20 rows regardless of depth
(page 1 and page 500 cost the same), and no COUNT is issued. Under
mutation it is stable: new rows above your position don't shift rows you
haven't read. The price: the ordering key must be unique and effectively
immutable (hence the `id` tiebreak â€” DRF's docs push you toward `created`
for exactly this reason), there is no "page 7 of 42", and users cannot jump
to an arbitrary page.

**Choice:** for the mobile app's infinite scroll over a mutating 10k-row
dataset â€” cursor, without hesitation; duplicates and skips are
user-visible bugs, and deep-scroll latency degrades exactly when engagement
is highest. Offset stays acceptable for small, slowly-changing,
page-numbered admin/reporting UIs. If clients already depend on page
numbers, that's a breaking API change to sequence, not just a paginator
swap.
